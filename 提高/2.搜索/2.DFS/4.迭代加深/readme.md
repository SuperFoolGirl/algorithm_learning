搜索时，搜索树可能会比较深

错误结点埋在很深的位置，导致资源浪费

但合法方案可能在一个比较浅的位置

迭代加深就适合这种问题：**搜索深度大，但答案比较浅**

**迭代加深基本思路：一层层搜**

和BFS有点像，但不同

迭代加深会设定一个层数上限max_depth，每次DFS时，如果当前层数超过max_depth，就直接返回

可以看作对于庞大的搜索空间，先划定一个小范围，在这个小范围内搜索解

如果搜不到，就把这个范围扩大一些(max_depth++)，再继续搜索

**与BFS的不同点：**

宽搜的空间复杂度是指数级别，迭代加深本质还是DFS，只研究一条路，**空间复杂度是线性级别的(与深度有关)**

这个优点是不可忽视的

**迭代加深的合理性**

迭代加深会**重复**搜索层数少的结点

这样一来，是否会造成“雪上加霜”？

从数学角度来看，并不会

假设搜索树是二叉树，答案在第10层，搜索前9层的消耗为1 + 2 + 4 + ... + 2^9 = 2^10 - 1

将搜索前n层的消耗视为2^n+1，那么搜索9次的消耗一共为：

2^2 + 2^3 + ... + 2^10 = 2^11

如果无迭代加深，正常搜索前10层，消耗为2^11 - 1

这里可以看到，在分支为2的情况下，重复搜索占用的资源:正常求解的资源 = 1:1，仅多用了一倍资源，数量级上没有差别

这还是分支数最少的二叉树的情况，如果分叉大，那么重复搜索占用资源的比例就更小了——10叉树时，冗余资源仅占10%

因此无需担心迭代加深产生负优化效果

**代码实现：**
在递归层数中维护一个参数depth，表示最大层数

main函数中，循环跑dfs，每次循环depth自增

while(!dfs(now,depth)) depth++, 直到depth增长到dfs可以搜到结果
