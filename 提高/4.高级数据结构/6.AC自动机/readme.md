### 介绍
AC自动机 = Trie + KMP，另外一种优化写法是Trie图——代码短一些，效率稍好，所以AC自动机一般会写成Trie图

搞清楚AC自动机是干嘛的：多模式匹配

模式匹配，指kmp中，用模式串p去对比文本串s，判断p是否在s中

而多模式匹配，就是给定多个模式串，判断它们是否在文本串s中

所谓“多个模式串”，其实就是字典树，根据文本串去跑字典树
#### 回忆Trie
字典树，每个结点预设26个子结点，表示26个字母；每个结点还可以存储一些信息，比如是否是单词的结尾，或者以这个结点为结尾的单词数量等

应当将字母视为**边权**而不是点权，边权图比点权图更直观，也更符合对于“路径”“字符串”的理解

沿着某一路径走到某结点时，若来路径为一个单词，那么应特殊标记该结点。进一步地，可对该结点维护更多需要的信息
#### 回忆KMP
这里现给出KMP的暴力做法：

应该这样理解这个过程：文本串在上，模式串在下，**用模式串去匹配文本串**；在文本串中枚举起点`t`，然后让`p[1]`对齐起点(前提是`p[1]==s[t]`)，开始一位位向后比较
```c++
// n为文本串长度，m为模式串长度
// kmp算法中，下标从1开始会更方便一些
for (int i = 1; i <= n; i++) {        // 枚举起点
    for (int j = 1; j <= m; j++) {    // 枚举模式串的下标
        if (s[i - j + 1] != t[j]) {   // 判断在当前起点下，文本串和模式串是否匹配
            break;
        }
    }
    if (j > m) {
        // 此时，匹配成功
    }
}
```
对于暴力做法，如果某处无法匹配，则起点`t`只会往后移动`1`位，导致效率过低

KMP的核心思想：若在`i`点处匹配失败，记录`s`最少要向后移动`x`位，才能为当前的`i`找到匹配点，且从s到`i`这一段也能顺利匹配

这也就是`next`数组的作用：`next[i]`表示在模式串p中，以`p[i]`结尾的后缀，能够匹配的从`p[1]`开始的最长非平凡前缀的长度(非平凡前缀，指不能将整个字符串视为“最大前后缀”)，也就是**前缀的最后一个字符的下标**

如`p="abcab"`，则`next[5]=2`，因为以`p[5]`结尾的后缀`"ab"`能够匹配的从`p[1]`开始的最长前缀是`"ab"`，长度为`2`

下面举一个例子来描述一下上面讲述的过程

设当前枚举到的起点是`t`，`p[1]`与`t`对齐开始比较，一直成功，直到`p[i+1]!=s[t+i]`处失败

此时，对于`p[1]~p[i]`这段模式串的前半部分，我们将其分为两部分：前半部分是**起点要向后移动的长度`x`**，后半部分是**`p`移动后保证`p[1]~p[i-x]`能匹配上`s[t+x]~s[t+i-1]`**

对于这个过程的核心理解还是在于，要视为“用模式串去匹配文本串”，千万不要搞反了

再回到上述的例子，目前`p[1]~p[i]`已经成功匹配了`s[t]~s[t+i-1]`，而`p[i+1]`与`s[t+i]`处失败了，我们让起点后移`x`

显然，移动的`x`越小越好，这样才能保证正确性；也就是说，对于`p[1]~p[i]`，**我们要求前半段最小，那么后半段就要最大**

回到`next`数组，**`next[i]`记录的是以`p[i]`结尾的后缀能匹配从`p[1]`开始的前缀的最大长度**，这恰好与我们的“后半部分”吻合

设`next[i]=y`，那么则有`p[1]~p[y]==p[i-y+1]~p[i]==s[t]~s[t+y-1]==s[t+i-y]~s[t+i-1]`（因为`p[1]~p[i]`已经成功匹配了`s[t]~s[t+i-1]`）

那么很简单了，让起点移动到`t+i-y`处，就有`p[1]~p[y]==s[t+i-y]~s[t+i-1]`了——后半段完全匹配上了

**前半部分，即移动长度为`i-y`；后半部分，即匹配长度为`y`；二者相加恰好为`i`，也就是`p[1]~p[i]`的长度**

此时，对于`p`的索引是`i+1`，我们令`i=next[i]`，就可以使得“后半部分的长度为`next[i]=y`”，恰好符合要求

在移动过程中，我们只保证了移动后不影响`p[1]~p[y]`的匹配，**而对于`p[y+1]`，移动后并没有保证，需要继续比较`p[y+1]`与最初的失配`s[t+i]`**。若依然失配，则循环操作

因此，我们可以给定kmp另一个理解：当`p[i+1]`与`s[t+i]`失配时，我们**向前**在`p[1~i]`寻找一个**最靠后**的字符`p[front]`来**试图**匹配`s[t+i]`——**前提**是`p[1~front-1]`要满足匹配，而这一点由`next[i]`来保证。在满足这个前提下，`p[front]`若不能匹配上`s[t+i]`，那么就循环操作；若`front`已经移动到0了，说明没有任何一个字符能匹配上`s[t+i]`，只能让起点`t`后移一位了(跳出while循环)

从上边的表述也能看出来，“矛盾”是不变的，依然是`s[t+i]`；这也印证了“用`p`去匹配`s`”

**上面是基于1-index讨论的；最初学习时是基于0-index写的。区别就在于，失配点在`p[i]`（0-index）还是`p[i+1]`（1-index）**

简单画个图：
```
设相等最大前后缀=next，失配点为i
s: xxxxxxx(next)uxxx(next)x...
p:        (next)uxxx(next)i...
移动后：
s: xxxxxxx(next)xxxx(next)x...
p:                  (next)u...
u(p[y+1])并不一定能匹配上x(s[t+i])，因此需要继续比较
```

优化代码：
```c++
// 由于使用1-index，字符串就不用string了
bool query(char s[], char p[]) {
    // n为模式串长度，m为文本串长度
    int n = strlen(p), m = strlen(s);

    // 计算next数组，也可以视为一种kmp：比较p的前缀和后缀是否相同
    // i索引模式串的下标，从1开始；j索引最长前后缀的长度，从0开始
    // 根据定义，next[0](不存在) = next[1](平凡前缀) = 0
    for (int i = 2, j = 0; i <= n; i++) {
        while (j && p[i] != p[j + 1]) {
            j = next[j];
        }
        if (p[i] == p[j + 1]) {
            j++;
        }
        // 此时，说明前缀p[1]~p[j]能匹配上后缀p[i-j+1]~p[i]，长度为j
        // 由于1-index，前缀的末尾下标j恰好就是长度
        // 注意j是可以跨越字符串中点的
        next[i] = j;
    }

    // kmp
    // i索引文本串下标，j索引模式串下标，原则上都从1开始
    // 这里由于一直用j+1来与s[i]比较，所以j的初始值为0
    for (int i = 1; j = 0; i <= m; i++) {
        // 若j+1处失配，令j=next[j]，试图向前寻找能够匹配上s[i]的p[front]，前提是p[1]~p[front-1]能够匹配
        // 若找不到，j=0跳出循环，寻找新的起点
        while (j && s[i] != p[j + 1]) {
            j = next[j];
        }
        if (s[i] == p[j + 1]) {
            j++;
        }
        // 当模式串下标索引到n时，说明模式串到末尾，匹配成功
        if (j == n) {
            return true;
        }
    }
    return false;
}
```

### AC自动机
普通KMP：对一维数组(字符串)建立next数组并进行模式匹配

AC自动机：对Trie建立next数组并进行模式匹配

AC自动机的next数组：Trie的每个节点u都有一个next数组，next[u]表示在Trie中，**以u结尾的后缀**能够匹配的**从root开始的最长前缀**所在的结点

注意，范围是整个Trie树，不只局限于u所在的root->u的路径上。普通KMP可以视为“只有一条链的Trie”

举例，给出root的两条路径：

```
root->s(1)->h(2)->e(3)
root->h(4)->e(5)
此时，有next[3]=5，next[2]=4
```

Trie的结点从0开始索引，0代表根结点；因此在形式上，对于无前缀的u总有next[u]=0，即指向根结点

回忆求next数组的过程，对于语句`while (j && p[i] != p[j + 1])`

由于上一轮末尾有`next[i] = j`，然后循环里`i`自增，因此上方语句可换成`while (next[i - 1] && p[i] != p[next[i - 1] + 1])`

这说明了，求next数组是依赖于**拓扑序**来递推的；对于Trie来说，拓扑序就是**层序遍历**

我们审视Trie可以发现，它自身就是一个DAG，直接宽搜就可以控制拓扑序了

#### 构建AC自动机的next数组
```c++
// 在普通KMP中，操作j++意味着按拓扑序走到下一个位置
// 但是在树中，可选择的路很多，不是使用++，而是按拓扑序转移到下一个位置
void build() {
    // 由于ne[0]=ne[1]=0
    // 对应到树中，就是根结点的next=0，根节点之下的第一层结点的next=0
    // 将下标对应到树的层数即可
    // 我们要从第二层开始求。这里必须通过第一层的结点去遍历第二层
    int hh = 0, tt = -1;
    for (int i = 0; i < 26; i++) {
        // 如果l1的结点有儿子，即存在对应的l2结点，那么才入队
        if (tr[0][i]) {
            q[++tt] = tr[0][i];
        }
    }

    while (hh <= tt) {
        // t <=> i-1, 即上一层的数据
        int t = q[hh++];

        // 遍历t的儿子
        // i <=> p[i], kmp中p[i]是一维字符串上的结点，i(字母)则是树上的结点(点权与边权可以互相转化)
        for (int i = 0; i < 26; i++) {
            // c <=> i, i是p的索引，tr存储结点的idx，则c是树的索引
            int c = tr[t][i];  // t的儿子c

            // 如果没有这个儿子，可以直接跳过
            if (!c) {
                continue;
            }

            // ne[t] <=> ne[i-1], 和kmp一样，j=ne[i-1]同时作为“模式串”的索引——也就是字典树结点移动的索引
            int j = ne[t];

            // !tr[j][i] <=> p[i]!=p[j+1]
            // 一维字符串上用p[i]和p[j+1]来比较，即“j能否走一步(+1)到i”
            // 对应到树上更直接了，tr[j][i]若为0，则j没有i这个儿子，那么j无法一步走到i
            // 走不到的话，就跳到树上另一条路径上去，这条路径(前缀)的结尾对应的结点是ne[j]
            while (j && !tr[j][i]) {
                j = ne[j];
            }

            if (tr[j][i]) {
                j = tr[j][i];  // 走到对应字符
            }

            // ne[c] <=> ne[i]
            ne[c] = j;
            q[++tt] = c;  // 别忘BFS入队
        }
    }
}
```
#### 模式匹配
```c++
// AC自动机中没有0-index/1-index之分，用层序代替1-index，实际数组从0开始即可
int query(char s[]) {
    int res = 0;
    for (int i = 0, j = 0; s[i]; i++) {
        int t = s[i] - 'a';
        while (j && !tr[j][t]) {
            j = ne[j];
        }
        if (tr[j][t]) {
            j = tr[j][t];
        }

        // 此时j匹配到了最深的位置，j出现过了，则ne[j]也递归地出现过了
        // 注意，在query过程中，我们并不知道“是否查询到了关键词(模式串)”
        // 不和kmp一样，有个 j == n 的判断
        // 这里的逻辑是，只要到达一个结点，别管是不是目标单词，我都统计；是的话正好，不是的话反证cnt=0也不影响答案。此时trie在跟着文本串跑，这期间只要统计了所有结点，就绝对不会漏下
        // 假设关键词有his和is，根据文本串，只有root-h-i-s，没有root-i-s，那is就会漏掉统计；因此，需要把所有ne链上的结点都进行统计
        // 当然，这部分逻辑可以通过增加节点信息来处理，就不多说了
        int p = j;
        while (p) {
            res += cnt[p];
            cnt[p] = 0;  // 如果不置0，可能会重复统计
            p = ne[p];
        }
    }
    return res;
}
```

### Trie图——AC自动机的优化
build的while循环虽然是线性，但常数较大；因此可以通过Trie图来优化，减少常数

如果失配，会沿着next指针不断向上跳；我们希望能够一步到位，不要跳这么多步
```c++
void build() {
    int hh = 0, tt = -1;
    for (int i = 0; i < 26; i++) {
        if (tr[0][i]) {
            q[++tt] = tr[0][i];
        }
    }

    while (hh <= tt) {
        int t = q[hh++];
        for (int i = 0; i < 26; i++) {
            int p = tr[t][i];
            // 从这里开始变动
            // trie图的逻辑有点像线索树。对于空节点，直接让他指向它应该跳到的位置
            // 比线索树更高超的是，通过合理修改ne数组，来实现跳跃的一步到位
            // 因此，可以视为tr和ne的协同优化
            if (!p) {
                // 如果没有儿子，往回跳到ne[t]结点并找i儿子
                // 以前的逻辑是：先跳到ne[t]，再找i儿子。这里把两步给合并了
                // 此时已经确定好字典了，不会再引入新结点，因此直接给tr[t][i]赋值即可
                tr[t][i] = tr[ne[t]][i];
            } else {
                // 如果有儿子，正常执行
                ne[p] = tr[ne[t]][i]; // 让ne直接指向更上层最终应该跳到的结果
                q[++tt] = p;
            }
        }
    }
}

int query(char s[]) {
    int res = 0;
    for (int i = 0, j = 0; s[i]; i++) {
        int t = s[i] - 'a';

        // 去掉循环，直接一步到位。tr和ne已经协同优化好了，直接大胆走
        j = tr[j][t];

        int p = j;
        while (p) {
            res += cnt[p];
            cnt[p] = 0;
            p = ne[p];
        }
    }
    return res;
}
```
